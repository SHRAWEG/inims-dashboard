# Internationalization (i18n)

## Section 1 — Two separate locale concerns

**UI locale** — language of the interface (buttons, labels, messages). Managed by `react-i18next`. Stored in `localStorage`.

**API locale** — language of the data from the backend (`?locale=en` or `?locale=ne`). Must always match UI locale. Passed as a query param on every translatable API call.

Both are managed together through `useLocale()` — never managed separately.

## Section 2 — Setup

```bash
npm install react-i18next i18next i18next-resources-to-backend
```

`src/i18n/config.ts`:
```ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import resourcesToBackend from 'i18next-resources-to-backend';

i18n
  .use(resourcesToBackend((language: string, namespace: string) =>
    import(`./locales/${language}/${namespace}.json`)
  ))
  .use(initReactI18next)
  .init({
    fallbackLng: 'en',
    defaultNS: 'common',
    supportedLngs: ['en', 'ne'],
    interpolation: { escapeValue: false },
  });

export default i18n;
```

## Section 3 — Locale files structure

```
src/i18n/locales/
  en/
    common.json    ← buttons, status labels, confirmations, errors
    auth.json      ← login page strings
    users.json
    sectors.json
    types.json
    <domain>.json
  ne/
    common.json
    auth.json
    users.json
    sectors.json
```

## Section 4 — `useLocale` hook

`src/hooks/use-locale.ts`:
```ts
export type SupportedLocale = 'en' | 'ne';

export function useLocale() {
  const { i18n } = useTranslation();

  const locale = i18n.language as SupportedLocale;

  const switchLocale = (newLocale: SupportedLocale) => {
    i18n.changeLanguage(newLocale);
    localStorage.setItem('locale', newLocale);
    // TanStack Query auto-refetches because locale is in query keys
  };

  return { locale, switchLocale };
}
```

## Section 5 — Locale in query keys

Locale must be part of every query key for translatable entities:

```ts
const { locale } = useLocale();

useQuery({
  queryKey: sectorKeys.list({ page, limit, search, locale }),
  queryFn: () => sectorsApi.getAll({ page, limit, search, locale }),
});
```

When user switches locale, all queries with locale in their key automatically refetch — no manual invalidation needed.

## Section 6 — Locale switcher component

`src/components/layout/locale-switcher.tsx`:
```tsx
export function LocaleSwitcher() {
  const { locale, switchLocale } = useLocale();

  return (
    <DropdownMenu>
      <DropdownMenuTrigger asChild>
        <Button variant="ghost" size="sm" className="gap-2">
          <Globe className="h-4 w-4" />
          {locale === 'en' ? 'English' : 'नेपाली'}
        </Button>
      </DropdownMenuTrigger>
      <DropdownMenuContent align="end">
        <DropdownMenuItem onClick={() => switchLocale('en')}>
          English {locale === 'en' && <Check className="ml-auto h-4 w-4" />}
        </DropdownMenuItem>
        <DropdownMenuItem onClick={() => switchLocale('ne')}>
          नेपाली {locale === 'ne' && <Check className="ml-auto h-4 w-4" />}
        </DropdownMenuItem>
      </DropdownMenuContent>
    </DropdownMenu>
  );
}
```

Always placed in the topbar.

## Section 7 — Rules

- Never hardcode any user-facing string — always `t('key')`
- `common` namespace for all shared strings — import with `useTranslation('common')`
- Each domain has its own namespace — `useTranslation('sectors')`
- Both `en` and `ne` JSON files must always be in sync — same keys, different values
- API locale always matches UI locale — managed by `useLocale()` hook
- Never use `i18n.language` directly in components — always through `useLocale()`
